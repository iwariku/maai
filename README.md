# Maai（間合い）

> **間合いを整える、現場のためのステータス可視化アプリ**

🔗 **デモサイト**:

メンバーのリアルタイムな状態（ステータス）をひと目で可視化するポートフォリオアプリです。

チーム内で **「休憩中の人に不要な連絡（電話など）をしてしまい、お互いの作業が中断され無駄なコミュニケーションが発生する」** という課題を解決することを最優先に設計しています。

アプリ名の **「間合い」** は、武道などで用いられる「相手とのちょうどよい距離・タイミング」を意味する言葉です。「相手の状態を尊重し、適切なタイミングで関わる」という本アプリの思想を体現しています。

---

## コンセプト

- **コミュニケーションコストの削減**: 誰が休憩中かを一目で把握できるようにし、不要な電話を未然に防ぐ
- **生産性の向上**: 作業中断を減らし、本来の業務に集中できる時間を最大化する

単なる状態管理ではなく、**「不要な作業中断を減らす」** という体験を中心に据えています。

---

## 背景（開発のきっかけ）

開発者自身がパティシエとして働いていた頃、店舗間の電話で次のような無駄なやり取りが頻繁に起きていました。

### 折り返し電話の応酬で無駄なコミュニケーションが発生する

1. 別店舗から「店長を呼んでください」と電話が入る
2. しかし店長が休憩中 → 受けた人が「折り返します」と応答
3. 店長が休憩から戻ったタイミングで連絡し、店長が折り返し電話をかける
4. 今度は **相手側の担当者が休憩中** → 再び「折り返します」
5. お互いの休憩タイミングを跨いで何往復もする…

「相手が今休憩中かどうか」が**事前に分かってさえいれば、この往復はそもそも発生しない**ものです。

### 休憩中なのに電話対応してしまう

幹部クラスの人が休憩中にも関わらず、別店舗から電話がかかってきて対応せざるを得ない、というケースもよくありました。しかも内容を聞いてみると **特別急ぎではない** ことが多く、本来であれば休憩明けに連絡すれば済む話です。

これは「相手のステータスが見えない」ことで、

- 発信側: 「今休憩中かもしれない」が判断できない → とりあえず電話してしまう
- 受信側: 鳴った電話を無視できない → 休憩を中断して対応してしまう

という両者にとって損な構造が生まれている状態です。

ステータスが可視化されていれば、**「休憩中だから今は連絡しないでおこう」** という判断が発信側で自然に働き、相手の休憩を守れます。

この実体験から、**リアルタイムなステータス、特に「休憩中」が一目でわかること** の価値が大きいと考え、本アプリを開発しました。

---

## 想定する展開先（業種・職種）

このアプリが解決する **「相手の状態が見えないことで起きる不要なコミュニケーション」** は、飲食業に限らず **複数拠点 × シフト制 × ノンデスクワーカー** で働く現場全般で同じ構造として現れます。

### 飲食業（原体験）

- 複数店舗を抱える飲食店で、店舗間電話が日常的に発生
- 店長や担当者が休憩中だと折り返しの応酬が起きる
- 緊急でない用件でも休憩中に電話対応してしまう

> ⚠️ 以下の **物流業 / 美容業** のケースは、開発者自身の直接体験ではなく、飲食業と同じ構造の問題が起きると考えた **仮説ベースの想定事例** です。

### 物流業（配送ドライバー）

- 運転中のドライバーに業務電話をかけてしまい、**安全面のリスク** + 受信側の負担になる
- ステータスで「運転中」が見えていれば、急ぎでない用件は運転後に連絡する判断が発信側で自然に働く

### 美容業（多店舗サロン）

- 複数店舗を展開するサロンで、別店舗から店長やスタイリストを呼ぶ電話がかかる
- 相手が施術中・休憩中だと折り返し対応になり、飲食業と同様の往復が起きる
- 接客中の対応強制で、目の前のお客様の体験にも影響しうる

### シフト制・ノンデスクワーカー全般（横串の特性）

上の3業種に共通する根本特性として、**シフト制で働くノンデスクワーカー** という働き方そのものが「ステータスが見えない」問題と直結します。

- **非同期な稼働**: シフトがバラバラで、誰が今動いていて誰が休憩時間かが常に変わる
- **PC に張り付かない**: テキストチャットを常時開ける環境ではないため、Slack 等の「在席アイコン」型ソリューションが機能しにくい
- **電話・口頭が主な連絡手段**: 結果として「今声をかけて大丈夫か」の判断材料が無いまま接触が発生する

→ ノンデスクワーカー向けの **「カードを見るだけで状態が分かる」シンプルな可視化UI** が、この層に最もフィットすると考えています。

### 共通する条件

- 拠点・チームが複数あり、拠点間の連絡が日常的に発生する
- 個人の状態（休憩・運転中・施術中など）が**相手から見えない**
- 連絡の用件が「緊急性は低いが、聞かないと判断できない」
- 働き手が **PC・テキストツール中心ではない**（ノンデスクワーカー）

→ これらに当てはまる現場ならどこにでも応用可能なソリューションです。

## 現状（MVP1）と今後

このバージョンはプロトタイプ（**MVP1**）として、以下の状態で公開しています。

- データはフロントエンドの状態管理のみ（ページリロードで初期化される）
- 5名のモックスタッフを初期データとして使用
- 「在席（デフォルト）」「休憩中」「休み」の3状態のみに絞り、**毎日のオン/オフ操作を不要**にする設計

### 今後追加予定の機能

- **休憩タイマー機能**: 「休憩中」へ切り替える際に休憩時間（例: 15分・30分・60分）を設定できる。タイマーが切れたら自動的に「出勤」状態に戻るため、**休憩後の戻し忘れを防ぎ、ステータスの正確性を保てる**
- **サーバー連携 / 認証・認可**: 実データでの運用
- **チーム単位での運用**: 組織・店舗単位でのスコープ管理

---

## ゴール

「誰が休憩中か」を極めてシンプルに可視化することで、チーム内の不要な作業中断を減らし、業務効率化とストレスのないコミュニケーションを実現することを目的としています。

---

## このプロダクトについて

このプロダクトは、**ノンデスクワーカー向けの社内コミュニケーション / エンゲージメント領域**（代表例: [株式会社スタメンの TUNAG](https://stmn.co.jp/)）への **新規機能提案を想定したポートフォリオ作品** として開発しました。

既存のエンゲージメントツールが「組織のつながりを強める」方向に強みを持つ一方で、本プロダクトは **「現場で本当に必要な、シフト制ワーカーのための小さな可視化機能」** という補完的な切り口で価値を提供することを意図しています。
